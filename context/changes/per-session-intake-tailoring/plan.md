# Per-Session Intake Tailoring (S-02) Implementation Plan

## Overview

At the start of every new session, capture three per-session inputs — **knowledge level for this material** (5-point scale), **learning goal** (short free text), and **available time** (bucket: ~15 / ~30 / ~60 min) — store them as typed columns on the `sessions` row, and make them **measurably bias** the generated session through two mechanisms: (a) injecting the values into the generation prompt, and (b) driving the number/depth of theory steps and MCQs from a deterministic sizing map. Correctness is proven by a Vitest unit test over the pure prompt-builder and sizing map.

This is the per-session half of FR-006. The profile-bio half (FR-005) is S-03 and is explicitly out of scope here — the generation signature is designed so bio can be added later without reshaping it.

## Current State Analysis

The S-01 session loop is live in production (`first-grounded-session`, archived). Today:

- **Flow** (`src/pages/sessions/new.astro:18` → `src/components/session/NewSessionForm.tsx`): pick file → parse text in the browser (`parseFile`) → `POST /api/sessions` with `FormData{file, extractedText}` → `generateSession(extractedText)` → persist session/material/theory/exercises → redirect to `/sessions/[id]`. **No intake is captured anywhere.**
- **Generation** (`src/lib/services/generation/generate.ts:65`): `generateSession(sourceText: string)` builds a fixed prompt via `buildMessages(source)` and validates the response against `GeneratedSessionSchema`. The prompt depends only on the source text.
- **Bounds are hard-coded** (`src/lib/services/generation/schema.ts:11-16`): `THEORY_MIN=3`, `THEORY_MAX=5`, `MCQ_COUNT=5`. The file already carries the marker `// intake-driven sizing is S-02` — this plan cashes that in.
- **`sessions` table** (`supabase/migrations/20260528202720_domain_schema_rls_baseline.sql:23-50`): `id, user_id, status, score, title, created_at, updated_at, completed_at`. F-01's four-policy RLS template is in place. **No intake columns.**
- **DB types**: `src/db/database.types.ts` is generated by `supabase gen types` and committed (Prettier-formatted).
- **No JS/TS test runner** exists — `package.json` scripts are `dev, build, preview, astro, lint, lint:fix, format`. The only tests are pgTAP (`supabase test db`). Adding the unit test means introducing **Vitest** (this repo's first JS unit test).

### Key Discoveries:

- `buildMessages(sourceText)` (`generate.ts:24-45`) is a **pure function** — ideal for a deterministic prompt-level test (resolves the roadmap's S-02 verification unknown).
- `GeneratedSessionSchema` (`schema.ts:41-45`) hard-codes counts; tailoring sizing means making the bounds a function of intake rather than module constants.
- The generation retry loop (`generate.ts:78-122`) catches API/JSON/schema/grounding failures and rebuilds nothing per attempt — `messages` is built once. Dynamic sizing must be computed **before** the loop and baked into both the prompt and the validation schema so both attempts agree.
- The API route (`src/pages/api/sessions/index.ts`) already re-validates the upload server-side (defense-in-depth) and generates **before** any DB write — intake validation and persistence slot cleanly into that existing structure.
- Lesson `context/foundation/lessons.md` — "Lazy-import browser-only libraries in SSR'd islands." The intake UI adds only plain form controls to the existing island; no browser-only lib is introduced, so the lesson is satisfied by default. Keep it in mind if any picker pulls a heavy dependency.

## Desired End State

A learner on `/sessions/new` must choose a knowledge level, type a learning goal, and pick a time budget before "Start session" is enabled. Those values are persisted on the session row and visibly shape the output: a 15-min "novice" session yields fewer, shallower theory steps and fewer MCQs than a 60-min "advanced" session, and the goal text steers topic focus. A committed Vitest test proves the built prompt contains the intake values and that the sizing bounds differ across buckets/levels.

**Verification:** `npm run test` (new) passes; `npm run build`, `npm run lint`, and `npx supabase test db` stay green; manual walkthrough shows differently-sized sessions for different intake combinations, with the persisted row carrying the three values.

## What We're NOT Doing

- **No conversational intake.** FR-018 prefers a 2–3 turn chat; we ship a structured form as a conscious MVP scoping call (top blocker = time). Conversational is a documented fast-follow, not part of this slice.
- **No profile bio / FR-005 / bio half of FR-006** — that is S-03. The `generateSession` signature leaves room for it but does not implement it.
- **No mid-session editing of intake params** — PRD non-goal; params are locked at session start.
- **No backfill** of intake onto pre-existing (S-01) sessions — new columns are nullable; historical sessions predate intake and are not re-tailored.
- **No model-output-quality assertions** in tests — we verify the prompt and sizing deterministically; output quality stays a manual check.
- **No change to the grounding/citation mechanism** from S-01.

## Implementation Approach

Build bottom-up so each phase rests on a committed contract: schema + a pure sizing map first, then generation consumes it, then the API wires inputs through, then the UI collects them, and finally the test pins the behavior. The sizing map is the linchpin — a single pure module that both generation (for bounds + prompt) and the test (for assertions) import, so there is one source of truth for "what intake produces what shape."

## Critical Implementation Details

- **State sequencing in generation:** compute the sizing target from intake **once, before** the retry loop in `generateSession`, and use it to build both the prompt (`buildMessages`) and the validation schema factory. If sizing were recomputed or the schema stayed static, a retry could validate against different bounds than the prompt requested. Bounds in → prompt and schema out, together.
- **Migration safety:** the three new columns must be **nullable** (or have defaults) — `sessions` already holds production rows from S-01 with no intake. Going forward the API always supplies all three; nullability only covers historical rows.

## Phase 1: Schema & intake foundation

### Overview

Add the typed intake columns to `sessions`, regenerate DB types, and create the shared intake types + the pure sizing-map module that downstream phases and the test all import.

### Changes Required:

#### 1. Migration — intake columns on `sessions`

**File**: `supabase/migrations/<timestamp>_session_intake_params.sql`

**Intent**: Persist the three per-session intake values on the session aggregate, typed and constrained, reusing F-01's existing RLS (no new policies needed — they live on `sessions` already).

**Contract**: `alter table public.sessions add column` ×3:
- `knowledge_level text` with `check (knowledge_level in ('novice','beginner','intermediate','advanced','expert'))`, nullable.
- `learning_goal text` with `check (char_length(learning_goal) <= 280)`, nullable.
- `time_budget_minutes smallint` with `check (time_budget_minutes in (15,30,60))`, nullable.
All nullable so existing rows remain valid; no backfill. No RLS changes (policies already cover all columns of `sessions`).

#### 2. Regenerate DB types

**File**: `src/db/database.types.ts`

**Intent**: Reflect the new columns in the committed generated types so the API layer is type-safe.

**Contract**: Run `npx supabase gen types typescript --local`, then Prettier-format (matches the committed style — the file is Prettier-formatted, not raw generator output). `sessions.Row`/`Insert`/`Update` gain the three fields.

#### 3. Shared intake types

**File**: `src/types.ts`

**Intent**: One canonical definition of the intake shape used by the service, API, and UI.

**Contract**: Export a `KnowledgeLevel` union (`'novice' | 'beginner' | 'intermediate' | 'advanced' | 'expert'`), a `TimeBudget` union (`15 | 30 | 60`), and a `SessionIntake` interface (`{ knowledgeLevel: KnowledgeLevel; learningGoal: string; timeBudgetMinutes: TimeBudget }`). Also export the ordered level list and time-bucket list for the UI to render and the API to validate against.

#### 4. Sizing-map module (pure)

**File**: `src/lib/services/generation/sizing.ts`

**Intent**: Deterministically translate intake into the structural target for a session: theory-step bounds, MCQ count, and a short natural-language depth/pacing directive injected into the prompt. Single source of truth for "intake → session shape," imported by generation and the test.

**Contract**: `export function sizeFromIntake(intake: SessionIntake): SessionSizing` where `SessionSizing = { theoryMin: number; theoryMax: number; mcqCount: number; depthGuidance: string }`. Time budget primarily drives counts (e.g. 15 → fewer theory + fewer MCQs; 60 → more); knowledge level shapes `depthGuidance` and nudges depth. Exact thresholds are the implementer's call but MUST be monotonic (more time ⇒ ≥ counts) and produce at least two distinct shapes across the three buckets so the test can assert variation. Pure, no I/O.

## Phase 2: Generation tailoring

### Overview

Make generation consume intake: inject the values into the prompt and replace the hard-coded bounds with sizing-map-driven dynamic bounds.

### Changes Required:

#### 1. Dynamic generation schema

**File**: `src/lib/services/generation/schema.ts`

**Intent**: Allow the theory/MCQ counts to be parameterized per session instead of fixed module constants, while keeping per-item schemas (TheoryStep, Mcq) unchanged.

**Contract**: Add a factory `makeGeneratedSessionSchema(sizing: SessionSizing)` returning a zod schema whose `theory` is `.min(sizing.theoryMin).max(sizing.theoryMax)` and `exercises` is `.length(sizing.mcqCount)`. Keep the existing constants as fallback defaults (used if generation is ever called without intake). `TheoryStepSchema`/`McqSchema` and the persisted `TheoryBody` type are unchanged.

#### 2. Intake-aware generation

**File**: `src/lib/services/generation/generate.ts`

**Intent**: Accept intake, compute sizing once, and thread it into both the prompt and validation so both retry attempts agree.

**Contract**: Signature becomes `generateSession(sourceText: string, intake: SessionIntake): Promise<GeneratedSession>`. Before the retry loop: `const sizing = sizeFromIntake(intake)`. `buildMessages(source, intake, sizing)` injects knowledge level, learning goal, time budget, `depthGuidance`, and the target counts into the system/user messages (the source-grounding rules from S-01 stay intact and authoritative). Validate with `makeGeneratedSessionSchema(sizing)` instead of the static schema. The citation-grounding check is unchanged.

## Testing Strategy

### Unit Tests (Vitest — new):
- `sizeFromIntake`: monotonic counts across time buckets; distinct shapes; level-dependent `depthGuidance`.
- `buildMessages`: intake values + target counts present in the prompt; different intake ⇒ different prompt.

### Integration / DB Tests (existing pgTAP):
- Existing `supabase test db` suite remains green (no RLS changes; new columns covered by existing session policies).
- Check constraints reject invalid level/time and over-long goal (manual Studio check; optionally a pgTAP assertion if cheap).

### Manual Testing Steps:
1. `/sessions/new`: confirm "Start session" is disabled until file + level + goal + time are all set.
2. Run two sessions on the same file: (15 min, novice) and (60 min, advanced); confirm the second has more/deeper theory steps and more MCQs.
3. Confirm the goal text influences topic focus.
4. In Studio, confirm the session row stores the three intake values.
5. Confirm a historical (S-01) session still opens (null intake tolerated).

## Performance Considerations

No new per-request server work beyond a few small form fields and the same single generation call. Larger time budgets request more theory/MCQs, modestly increasing generation tokens/latency — bounded by the 60-min bucket's caps. No impact on the Cloudflare Worker CPU budget (generation is an I/O-bound subrequest, as in S-01).

## Migration Notes

- New `sessions` columns are nullable; **no backfill**. Existing S-01 sessions keep `null` intake and render unaffected.
- Forward deploy: apply the migration to prod (`supabase db push`) **before** the code deploy, per the S-01 deploy ordering (schema-before-code). Paired down-migration only needed if rolling back across this boundary.
